## What shipped

`scripts/run_e2e_tests.py` — Tier 1 #1.4 e2e regression harness, adapted
from a teammate's testing system (`docs/external_reference/run_e2e_tests.py`,
~2,700 lines, 90+ scenarios). Translated to our LangGraph + TutorState
architecture; drives via `graph.invoke()` directly (same code path the
WebSocket route runs through, no uvicorn dependency).

### Stage 1 framework

- **Assertion library** (architecture-agnostic) ported from teammate's:
  `has_question`, `at_most_two_questions`, `non_empty`, `no_meta_leak`,
  `no_concept_leak(concept)` (with discriminating-stem matching + word
  boundaries + generic-word skip list), `does_not_confirm_wrong_answer`,
  `not_jailbroken`, `reveals_concept`, `looks_like_close_or_assessment`,
  `looks_like_redirect`.
- **Post-scenario state assertions** (our state machine):
  `reach_coverage_at_least(min)`, `reach_path_one_of({...})`,
  `topic_confirmed`, `phase_in({...})`, `no_interventions`.
- **`Scenario` + `TurnSpec` dataclasses** for declarative scenario
  definition.
- **Async driver loop** (`run_scenario`): runs rapport phase, then per
  turn appends student message → invokes graph → captures tutor reply →
  runs assertions → snapshots state. Captures pre/post state snapshot,
  per-turn elapsed, per-turn assert results. Bails gracefully on
  exception.
- **Markdown reporter** (`render_report`): summary table + per-scenario
  detail with PASS/FAIL per assertion, tutor message preview,
  per-turn state snapshot, post-scenario state checks.
- **Per-scenario JSON dumps** for forensics:
  `data/artifacts/e2e/<timestamp>/<sid>_<cat>.json`.

### CLI

```bash
SOKRATIC_RETRIEVER=chunks .venv/bin/python scripts/run_e2e_tests.py
SOKRATIC_RETRIEVER=chunks .venv/bin/python scripts/run_e2e_tests.py --only A1,B1
SOKRATIC_RETRIEVER=chunks .venv/bin/python scripts/run_e2e_tests.py --category A,F
```

Each scenario runs as `student_id="test_<sid>_<conv8>"` so transcripts
are visible alongside production sessions in the UI sidebar.

### Smoke test result

Ran `--only A1`. Framework works correctly end-to-end:
- Builds graph + retriever + memory
- Runs rapport + 3 turns
- Generates `report.md` and `A1_A.json`
- Reports clear PASS/FAIL per assertion + per-state-check

A1 failed — but for **two substantive bug findings** (not framework
issues), filed as follow-ups for Stage 1.5:

1. **Topic-card surfacing on specific heart query** — the broad query
   `"What part of the heart starts the heartbeat?"` triggered borderline-
   match cards instead of locking a topic. Cards shown were random
   (`Fluid Compartments / Muscles That Move the Feet and Toes / The
   Teeth`) suggesting `sample_diverse` fired instead of `sample_related`
   for this topic-engagement entry point. Same bug shape that drove
   the original `sample_related` introduction in round 1.
2. **Lock-anchor latched onto a sub-concept** — `locked_answer='pacemaker
   cells'` instead of canonical `'SA node'` / `'sinoatrial node'`. As a
   result, when the student literally said `"the SA node — the
   sinoatrial node"`, the reach gate (Step A) didn't match the locked
   answer's tokens and reach=False even on a correct answer.

Both are pre-existing bugs surfaced by the new harness — they're not
regressions from any code shipped today. The harness is the diagnostic
that revealed them, which is exactly what it was built for.
