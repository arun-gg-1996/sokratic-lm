## What was implemented

### `ingestion/chunk.py` — fully implemented

Takes the 5,278 raw elements from `raw_elements_ot.jsonl` and produces semantic
chunks saved to `data/processed/chunks_ot.jsonl`.

#### Key decisions

**Element type routing**
- Tables → 1 chunk each, no splitting. `element_type="table"`.
- Figure captions → 1 chunk each, no splitting. `element_type="figure_caption"`.
- Paragraphs → section-level grouping + semantic splitting.

**Section-level grouping (not subsection-level)**

`_para_group_key` returns `(chapter_num, section_num)`, not `(chapter_num,
section_num, subsection_title)`. Section-level groups have median ~22 elements
(vs. median ~3 for subsection groups), giving the splitter enough material to
combine short adjacent paragraphs and split only at genuine topic shifts.
This produced 204 groups total (after oversized sub-splits).

**SemanticSplitterNodeParser configuration**
- Model: `text-embedding-3-small` (OpenAI), `embed_batch_size=100`
- `breakpoint_percentile_threshold=95` — only splits at the 5% most dissimilar
  sentence transitions (conservative). Threshold of 75 produced 5,513 chunks
  on the first run (too many); 95 yielded 1,886 (within Gate 1 range).
- `buffer_size=1`

**Per-chunk metadata recovery (subsection_title and page)**

`subsection_title` varies within a section, so it cannot live in `base_meta`.
Instead, element positions are tracked as `(start_char, end_char, page,
subsection_title)` tuples across the joined document string. After each node
is emitted, `_find_meta_for_chunk` locates the chunk's start in the joined text
via substring search and maps it back to the source element's range, recovering
both `page` and `subsection_title` per chunk.

**Oversized group splitting**

Groups exceeding `MAX_GROUP_CHARS=30_000` chars are split at element boundaries
to keep individual OpenAI embedding calls manageable. This is why there are 204
groups for what would otherwise be ~197 section-level groups.

**Short chunk filtering**

`MIN_CHUNK_CHARS=100` — chunks shorter than 100 characters are discarded both
before and after splitting.

## Issues encountered and fixes

**Issue 1: First run produced 5,513 chunks (Gate 1 requires ≤ 3,000)**

- Root cause: `breakpoint_percentile_threshold=75` splits 25% of all sentence
  transitions. Combined with 1,041 tiny subsection-level groups (median 3
  elements, 290 single-element groups), the splitter was splitting nearly every
  element further rather than combining short adjacent paragraphs.
- Fix (two-part):
  1. Changed `_para_group_key` to return `(chapter_num, section_num)` only —
     section-level grouping produces 197 groups with median ~22 elements.
  2. Raised `breakpoint_percentile_threshold` from 75 → 95 — only the 5% most
     dissimilar transitions are split.
- Empirical projection before re-run (20-group sample): ~0.29 chunks/element
  ratio → projected 2,044 total; actual result: 1,886. ✓

**Issue 2: `subsection_title` and `page` recovery across section-level groups**

- Because groups now span multiple subsections, `base_meta` cannot carry a
  single `subsection_title`. Metadata must be recovered per-chunk.
- Fix: element_positions tracking `(start, end, page, subsection_title)` per
  element in the joined document; `_find_meta_for_chunk` does substring lookup
  to recover both values.
