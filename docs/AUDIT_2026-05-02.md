## L1. Data Layer Split

**SQL (SQLite, raw `sqlite3`, file-based, server-portable) owns:**
- All counts (sessions, strong/weak/unfinished tallies)
- Per-session metadata (thread_id, dates, locked topic, reach status, mastery tier, hint level final, turn count)
- Per-subsection mastery EWMA + last_outcome + attempt_count
- Full message transcripts (stored as separate JSON files at `data/student_state/sessions/{thread_id}.json`, path stored in DB)
- All structured / aggregation / exact-lookup queries

**mem0 (Qdrant `sokratic_memory`) owns ONLY:**
- `category=misconception` — concrete self-contained sentence per misconception event
- `category=learning_style` — concrete style observation per session

**Dropped from mem0 entirely** (move semantics to SQL):
- `session_summary`
- `topics_covered`
- `open_thread` (writer kept for stat counter; reader dropped; counter lives in SQL)

## L3. Mastery Rollup + UI Coloring

- Subsection score: existing EWMA = `0.6 × fresh + 0.4 × prior`. No change.
- Section / topic / chapter score: **mean of TOUCHED children** (untouched children excluded — no zero-fill).
- Coverage signal alongside score on every node: `N touched of M children` (so a section showing 80% based on 1 of 12 children doesn't mislead).
- Color thresholds: **green ≥ 0.75 / yellow 0.5–0.75 / red < 0.5 / grey untouched**.
- Per-session categorical `mastery_tier` (proficient/developing/needs_review/not_assessed) preserved unchanged for eval scoring.

## L6. mem0 Semantic Injection Points (3 total)

1. **At topic-lock time** (once per session, implicit pull): query mem0 with `query = locked_question + subsection`, filter `category in (misconception, learning_style) AND subsection_path = locked.subsection`, top-K=2. Inject into teacher's first turn prompt.

2. **At hint-advance** (every level bump 1→2 or 2→3, implicit pull): query mem0 with `query = locked_question + "what worked when stuck"`, filter `category=learning_style`, top-K=1. Inject as style cue into next hint generation.

3. **At memory_query intent** (P1-E, explicit handler): when student asks "what did we cover before?" mid-chat. Filter `category=session_summary`, top-K=3. **NOTE:** session_summary is now in SQL, not mem0 — so this handler reads SQL `sessions` table and (optionally) supplements with mem0 misconception recall for richer context.

**No mid-turn LLM tool-calling for mem0.** All injection is implicit at deterministic decision points (deterministic = good for eval reproducibility).

## L7. Hint Generation

- Always dynamically generated by teacher LLM with `hint_level + retrieved_chunks`.

## L9. Topic Mapper LLM (NEW — replaces 3-stage fuzz pipeline)

**Today vs plan — pipeline replacement explicit:**

```
TODAY (Nidhi's branch — 3 stages, fuzz fallback present):
  Student free-text
    → LLM intent classification (Haiku, normalize)
    → Retriever semantic vote (Qdrant top hits)
    → RapidFuzz fallback if semantic vote ambiguous

```

**RapidFuzz fallback is REMOVED entirely** in the plan. So is the semantic-vote intermediate stage. So are the fuzz thresholds (`STRONG_MIN=78`, `STRONG_GAP=10`, `BORDERLINE_MIN=50`). Single Haiku call is the only topic-resolution path post-rebuild.

This is the same removal flagged in L8 ("Rule (REMOVED entirely)") — re-stated here for implementation clarity. Codex's audit confirmed RapidFuzz is still in Nidhi's branch (`retrieval/topic_matcher.py:match` + `_score` + threshold constants). The plan pulls the entire 3-stage pipeline out and replaces with one Haiku call.



**Model:** Haiku (cheap classifier).
**Temperature:** 0.0 (classification, not generation).
**Output:** strict JSON, max_tokens long enough to never truncate.

**Inputs to the prompt:**
1. Student utterance (raw free-text).
2. Full TOC + per-leaf 1-line summary.
   - Source: join `data/topic_index.json` (363 entries) with `data/artifacts/raptor_subsection_summaries.jsonl` (549 entries) on (chapter, section, subsection).
3. Curated abbreviation list (~20–30 entries: CN VII, RCA, LAD, ATP, GFR, BMI, SA node, AV node, etc.) — **explicitly framed as "common shortcuts students use, generalize from your own medical knowledge — not exhaustive"**.

**Output schema:**
```json
{
  "verdict": "strong" | "borderline" | "none",
  "confidence": 0.0,
  "student_intent": "topic_request" | "topic_request_with_question",
  "deferred_question": "..." | null,
  "top_matches": [
    {"path": "Chapter 11 > Section > Subsection", "confidence": 0.0, "rationale": "..."},
    ... up to 3
  ]
}
```

**Confidence thresholds for routing:**
- `verdict=strong`, confidence ≥ 0.85 → `lock_anchors_call` → coverage gate
- `verdict=borderline`, confidence 0.7 – 0.85 → confirm-and-lock UX (Gap 1)
- `verdict=borderline`, confidence 0.5 – 0.7 → cards from `top_matches`
- `verdict=none`, confidence < 0.5 → `_prelock_refuse_call` intro + `sample_diverse` cards

## L26. Honest-Tone Principle for Session-Close Messages

All session-close messages (turn-25 graceful close + memory_update closing message) must reflect actual session quality. No generic praise.

- Reached cleanly → acknowledge skill, name what was done well specifically
- Reached with heavy scaffolding → acknowledge effort, name the scaffolding pattern honestly ("you got there with the analogy; the underlying mechanism still needs work")
- Didn't reach → name what was managed + what's left for next time, no consolation fluff
- **Never** say "great job" / "excellent work" / "well done" by default — those phrases require evidence

Per-turn anti-sycophancy already covered by Nidhi's `haiku_sycophancy_check`. This is the close-message addition.

Temperature for close-message generation: 0.0 (honesty signal must be stable, not creative).

## L27. RAG Architecture (Decisions from Q11–Q14)

**Two distinct retrieval concepts, separate buffers (Q11 revised → b):**

| Concept | When | Buffer | Mutation |
|---|---|---|---|
| Anchor retrieval | Once at lock time | `state.anchor_chunks` | Immutable post-lock |
| Tangent retrieval | Every tutoring turn | `state.tangent_chunks` | Per-turn scratch (replaced each turn) |

Both passed to teacher for current turn's draft. Anchor stays canonical for hint generation + reach gate. Tangent never updates locked anchors.

**Query construction (Q12 → b):**
- Anchor query = the resolved topic string (from `topic_mapper_llm`'s output `path` or the card-pick `path`).
- Tangent query = the actual student utterance for that turn.
- No blending. No deterministic patches.

**Tangent retrieval params (Q13 → always-fire + conditional inject):**
- Always retrieves fresh chunks every tutoring turn (cheap — local Qdrant).
- top_k = 3–5 (small enough to not bloat context).
- No section filter (broad recall — sibling subsections + cross-system links allowed).
- **Conditionally injected into teacher prompt** based on `_setup_call`'s extended classifier — when `student_state == "tangent_question"`, teacher gets anchor + tangent chunks; otherwise anchor only.
- No new LLM call (extends existing `_setup_call` enum by one value).
- Drop the existing `_exploration_judge` — folded into `_setup_call`.

**Anchor retrieval params (Q14 anchor):**
- top_k = 12 (unchanged).

## L35. Mid-Session Reconnect — DROPPED

**Decision (2026-05-02):** no mid-session reconnect. Browser close = session ends, period. Per Codex round-3 fix #2, wording aligned with L21:

  AND started_at < (now() - interval '1 hour')

This is read-only analytics — no row mutation, no automatic status flip. The 1-hour grace window prevents a session that's actively being used from being miscategorized as abandoned.

## L43. Architectural Choice — Option C (vs Option B)

Going with **Option C: rules + Dean planning + Haiku catcher.** No structural redaction.

- Teacher sees full chunks (with answer visible)
- Dean's TurnPlan provides `forbidden_terms` Teacher must avoid
- Parallel Haiku self-checks catch leaks after Teacher drafts
- Expected leak rate: ~3% (today's measured baseline, with stack improvements)

**Option B (redaction) deferred to Phase 6** if measured leak rate post-rebuild proves too high. TurnPlan already includes `apply_redaction: bool` field defaulted to `false` so Phase 6 swap is additive.

Reasoning:
- Synonym enumeration is brittle (anatomy has many short common-word answers — "node", "fiber", "cell" — Option B would over-redact unrelated mentions)
- Voice quality with redacted context is unmeasured; can't A/B before paper deadline
- Empirical "leak rate dropped from X% to Y% via this stack" is a stronger paper claim than binary "cannot leak"

## L44. Pre-Flight Haiku Layer (3 parallel checks)

Before Dean runs on each tutoring turn, three parallel Haiku classifiers fire in pre-flight:

1. `haiku_help_abuse` — catches "just tell me", "idk" stalling patterns
2. `haiku_off_domain` — catches full off-domain content (#1 only — "what's your favorite movie", chitchat, jailbreak attempts; NOT on-domain tangents)
3. `haiku_deflection` — catches session-end requests ("can we stop?", "let's finish", "I'm done")

Cost: ~$0.0001 per Haiku × 3 parallel = ~$0.0003 per turn (vs Sonnet Dean's ~$0.038 per turn skipped).

**If any pre-flight fires → Dean SKIPPED for this turn.** Action depends on which fired (see L55, L56, L58).

**If none fire → Dean runs normally (L46+).**

## L45. Pre-Flight Redirect Rendering — Teacher (LLM) NOT Templated

When pre-flight fires, the redirect/nudge/confirm message is written by **Teacher** via the standard TurnPlan contract — NOT a templated string.

Rationale: templated replies feel canned and break the conversational tone consistency. Teacher writes the redirect using a TurnPlan with:
- `mode = "redirect" | "nudge" | "confirm_end"` (per pre-flight category — selects Teacher prompt path)
- `tone = "neutral" | "firm"` (emotional register — defaults to "neutral" for redirects, "firm" for repeated catches)
- `apply_redaction = false` (no answer to redact yet)
- Standard shape_spec (max sentences, exactly one question or directive)

Cheaper than full Dean+Teacher loop (Dean skipped), still personalized.

## L46. TurnPlan Schema (Dean→Teacher Contract)

Single-source contract from Dean to Teacher. Locked field spec:

```typescript
TurnPlan {
  scenario: str               // 1-line summary of student state
                              // ("partial answer, needs sodium-pump nudge")
  hint_text: str              // the actual hint Dean wants Teacher to render
  permitted_terms: list[str]  // vocab Teacher can lean on
  forbidden_terms: list[str]  // vocab Teacher must avoid
                              // (locked_answer + aliases + derived synonyms)
  // Codex round-2 fix: split mode (situation/phase) from tone (emotional register)
  mode: str                   // SITUATION/PHASE — what KIND of message Teacher renders
                              // "socratic" | "clinical" | "rapport" | "opt_in" | "redirect"
                              //   | "nudge" | "confirm_end" | "honest_close"
                              // (used by single Teacher entry point per L49 to pick prompt mode)
  tone: str                   // EMOTIONAL REGISTER — how the message FEELS
                              // "encouraging" | "firm" | "neutral" | "honest"
                              // (used by Teacher prompt to shape phrasing; orthogonal to mode)
  shape_spec: dict            // {max_sentences: 4, exactly_one_question: true}
  carryover_notes: str        // mem0-derived prior-session context
                              // (misconception/learning_style relevant to this turn)
  hint_suggestions: list[str] // soft hint bank from lock time, Dean's optional
                              // reuse pool (may be empty; informational only)
  apply_redaction: bool       // ALWAYS false in our design (Option C)
                              // reserved for future Phase 6 (Option B)
  student_reached_answer: bool // INFORMATIONAL ONLY, NOT authoritative
                              // Authority is reach gate (Step A + Step B per L53)
  image_context: dict | null  // VLM JSON output (per L77), present when session was image-initiated:
                              //   {identified_structures, image_type, description, best_topic_guess}
                              // Teacher uses description text to ground responses (no per-turn vision).
                              // Lock prompt receives this to generate identification-style locked_question.
                              // null for text-only sessions.
  clinical_scenario: str | null   // Patient/case scenario text (per L74). Set when transitioning to
                                  // clinical phase via Dean's planning call. null in tutoring phase.
  clinical_target: str | null     // The expected reasoning/answer for the clinical scenario.
                                  // null in tutoring phase.
}
```

If TurnPlan parse fails → fallback: Dean re-emits with stricter prompt; if 2nd parse fails, ship a minimal TurnPlan with only `scenario`, `hint_text`, `tone="neutral"`, `shape_spec={max_sentences:3, exactly_one_question:true}`.

## L47. Per-Turn Hint Generation + Soft Hint Suggestions

**Hints are generated per-turn by Dean, not pre-baked.**

- At lock time: Dean optionally generates a small "hint suggestions bank" (~5 angles for the locked subsection, $~0.005 cost). Stored as `state.hint_suggestions: list[str]`.
- Per-turn: Dean's `_setup_call` decides whether to reuse a bank suggestion verbatim OR write fresh. Bank entries are SOFT — Dean can ignore them entirely if student state suggests a different angle.
- TurnPlan's `hint_suggestions` field carries the remaining bank entries forward each turn (so Dean always sees what's available).

Why per-turn over pre-baked: pre-baked hints don't know what student said in turns 1-N. Per-turn Dean can respond to actual student state. Bank exists as fallback / scaffolding for Dean's own thinking, not as authoritative hint plan.

**Drop today's `hint_plan` skeleton** that pre-bakes 3 hints at lock time and uses them sequentially. Replaced by soft suggestions bank.

## L48. Teacher Self-Policing — 4 Parallel Haiku Checks

**Project-spec compliance note (CSE 635 P3):** the spec requires *"strictly forbidden from providing a direct definition or answer in the first two turns."* Our system satisfies this naturally via hint progression — `hint_level` starts at 0 (no hint), advances to 1 (oblique) after first incorrect attempt, and only reaches 3 (most direct) after multiple wrong turns. So the earliest a "direct" hint can fire is turn 4+ in practice. `haiku_leak_check` provides a backstop on top of this. No additional explicit "turn ≤ 2 = no direct answer" guard needed.



After Teacher drafts, four cheap Haiku checks fire in parallel:

1. `haiku_leak_check` — covers EULER `no_reveal` (no answer disclosure, no unique-feature reveals)
2. `haiku_sycophancy_check` — covers anti-sycophancy + appropriate tone match to TurnPlan
3. `haiku_shape_check` — five sub-checks (per L59)
4. `haiku_pedagogy_check` — covers EULER `relevance` + `helpful` per L60

All return structured output:
```typescript
{
  pass: bool,
  reason: str,    // 1-line explanation if pass=false
  evidence: str   // specific text fragment that triggered failure
}
```

If all pass → ship draft. If any fail → retry Teacher with feedback (L62).

Latency: ~1 max wall-clock unit (4 parallel calls run concurrently). Cost: ~$0.0008 per turn (4× Haiku).

## L49. Single Teacher Entry Point

Teacher's 4 functions today (`draft_socratic`, `draft_clinical`, `draft_clinical_opt_in`, `draft_rapport`) collapse into ONE function:

```python
teacher.draft(turn_plan: TurnPlan, chunks: list[Chunk], history: list[Message]) -> str
```

TurnPlan's `mode` field selects which prompt path Teacher uses (per Codex round-3 fix #1):
- `socratic` → tutoring scaffolds with hint_text
- `clinical` → clinical scenario rendering
- `rapport` → opener phrasing
- `opt_in` → clinical opt-in question
- `redirect` / `nudge` → pre-flight responses
- `confirm_end` → deflection / off-domain end confirmations
- `honest_close` → session-end summaries

TurnPlan's `tone` field is orthogonal — selects emotional register (`encouraging` / `firm` / `neutral` / `honest`) within the chosen mode. So the same `mode="redirect"` can have `tone="neutral"` (gentle) or `tone="firm"` (after multiple catches), shaping phrasing without changing the prompt path.

One prompt template per mode, modulated by tone variable. Consistent voice across phases. Reduces drift.

## L50. Retry Cap + Escalation (revised per Codex fix #5 — never ship a known leak)

Hard limit on Teacher's draft attempts:

```
attempt 1: teacher.draft(turn_plan, ...)
  parallel haiku × 4 → if all pass, ship

attempt 2: teacher.draft(turn_plan, ..., prior_attempt=draft_1, prior_failures=[...])
  parallel haiku × 4 → if all pass, ship

attempt 3: teacher.draft(turn_plan, ..., prior_attempts=[d1,d2], prior_failures=[...])
  parallel haiku × 4 → if all pass, ship

all 3 fail → Dean re-plans ONCE:
  new_turn_plan = dean.replan(original_turn_plan, ..., prior_attempts=[d1,d2,d3], prior_failures=[...])
  draft_4 = teacher.draft(new_turn_plan, ...)
  parallel haiku × 4 →
    if all pass → ship draft_4
    if leak_check FAILS → DO NOT SHIP draft_4 — fall back to safe generic probe (below)
    if other check fails (sycophancy / shape / pedagogy) → ship draft_4 anyway (less critical than leak)
```

**Critical safety rule (Codex fix #5):** never ship a draft that fails `haiku_leak_check`. Leaks are the highest-stakes failure mode for a Socratic tutor — better to send a generic probe than to leak the answer.

**Safe generic probe fallback** (deterministic, templated, no LLM call, guaranteed non-leak):

```
"Let me reframe — what aspect of this would you like to think through more carefully?
Pick a piece you're less sure about and we can explore it together."
```

Used only when leak_check still fails after 3 retries + 1 Dean re-plan. Logged to `state.debug.turn_trace` with a `leak_cap_fallback_fired = True` flag for eval. Never increments hint level (the turn is effectively wasted; student gets no new content but no leak either).

Bounded latency, bounded cost. Worst case per turn: 4 Teacher drafts + 4×4=16 Haiku checks + 1 Dean re-plan + maybe 1 fallback render = ~$0.05 worst case. Average case: 1 Teacher draft + 4 Haiku = ~$0.005.

Hard timeout per turn: 30 seconds wall-clock. If exceeded, fall back to safe generic probe (same fallback path).

## L55. Help-Abuse Counter (Strike-Based, Hint Advance at 4)

Why: prevents hint farming (4 idks needed before next hint) AND eventually unblocks genuinely-stuck students.

## L56. Off-Topic Categories + Off-Domain Counter (4-Strike Termination)

Three categories of "not on locked topic":

1. **Full off-domain (#1)** — "what's your favorite movie?", chitchat, jailbreak. Caught by `haiku_off_domain` in pre-flight.
2. **On-domain but different subsection (#2)** — anatomy-related but not the locked subsection. Treated as `tangent_question` (collapsed into #3).
3. **On-subsection tangent (#3)** — within or adjacent to locked subsection. Classified by `_setup_call` as `tangent_question` → tangent retrieval fires (L27).

**Off_topic counter applies ONLY to #1.** #2 and #3 are not penalized.

Off_topic counter behavior:
- Pre-flight catches off-domain → Teacher writes nudge (escalating tone: turn 1 gentle, turn 2 firmer, turn 3 direct) → counter increments
- At strike 4 (consecutive): graceful end. Teacher writes honest closing message naming chapter > section > subsection. Session marked `mastery_tier=not_assessed`, `reach_status=null`, status flag = "ended_off_domain".
- No mid-session topic switching. Student is told to start fresh from My Mastery.

## L57. Tangent_Question Detection in `_setup_call`

`_setup_call` enum extended:
```
correct | partial_correct | incorrect | low_effort | question | tangent_question
```

Distinguishing `question` vs `tangent_question`:
- `question` = student asks clarification about the locked anchor or current subsection — answered from anchor chunks (no fresh retrieval needed)
- `tangent_question` = student asks about something within OR adjacent to the subsection that anchor chunks may not fully cover — triggers tangent retrieval

**Prompt instruction:** "When in doubt, classify as `tangent_question` so Teacher has richer context. Tangent retrieval is cheap and never harmful — anchor chunks always remain available."

Bias toward fail-safe classification: false positive (firing tangent retrieval when not needed) costs ~50ms; false negative (missing a tangent) means Teacher might lack relevant chunks.

## L58. Deflection State Handling

Detection: 3rd parallel Haiku in pre-flight (`haiku_deflection`). Catches:
- Explicit: "let's stop", "I want to end this", "can we finish?"
- Implicit: "I'm tired, can we stop?" / "this is taking too long, let's wrap up"

When detected:
- Dean SKIPPED for this turn
- Teacher writes confirm message via TurnPlan with `mode="confirm_end", tone="neutral"`
- UI renders YES/NO buttons (same component as L10 confirm-and-lock)
- Click YES → graceful end (mastery_tier reflects current state, status="ended_by_student")
- Click NO → continue normally (next student message routes through full pre-flight + Dean as usual)
- Type free-text → treat as a new student message, route through pre-flight (re-classified)

**No counter for deflection.** Each detection independently confirmed. If student keeps signaling "I'm tired" but says "no" to confirm, respect that. Re-confirming on every "I'm tired" feels naggy.

## L59. `haiku_shape_check` Responsibilities (5 sub-checks, all Haiku, no regex)

Five things the shape check verifies:

1. **Length** — draft ≤ `shape_spec.max_sentences`
2. **Single question** — exactly 1 question in the draft (when `shape_spec.exactly_one_question = true`)
3. **Banned prefixes** — doesn't start with "Great!", "Excellent!", "Perfect!", "You got it!", "Wonderful!", etc.
4. **Hint-level alignment** — directness consistent with current hint_level (level 1 oblique, level 3 very direct). Prompt receives a 1-line description per level + Dean's intended `hint_text` from TurnPlan for comparison.
5. **No repetition** — draft isn't substantively the same as the previous tutor turn's question. Receives prior 1-2 tutor questions for comparison.

All five run inside ONE Haiku call (single prompt with multiple verification instructions). NO regex anywhere.

Risk: #4 (hint-level alignment) is the hardest for Haiku. Mitigation: bias toward "pass" when ambiguous. If post-rebuild eval shows hint-level violations slipping through, swap shape_check to Sonnet (parallel becomes Sonnet for this slot, others stay Haiku).

## L60. `haiku_pedagogy_check` (NEW — covers EULER live)

New 4th parallel Haiku check, separate from shape:

- **EULER `relevance`** — does the draft address content within the locked subsection / locked anchor scope?
- **EULER `helpful`** — does the draft advance reasoning toward the answer (not just restate the question)?

Live per-turn checking of these EULER dimensions (not just offline scoring). Catches failures BEFORE student sees them. Same checks as offline scorer = consistent quality metric across live + eval.

Cost: ~$0.0002 extra per turn (one more Haiku in the parallel quartet).

## L63. memory_update Orchestration at Session End

Final orchestration when session ends (any path: reach, hint exhaust, turn limit, off-topic strike-4, deflection confirm).

**Ordering matters** (per Codex audit fix #1): closing message is composed + appended to messages BEFORE transcript persistence + memory extraction. Otherwise the transcript file wouldn't include the goodbye, and memory extraction wouldn't have it as context either.

```
1. Compose closing message via Teacher (TurnPlan with mode="honest_close",
   tone="honest", temperature 0 per L26+L28, metadata={"is_closing": True})
   — render to student AND append to state.messages.
2. Mark sessions.ended_at = now(), update all session fields
   (core_mastery_tier, clinical_mastery_tier, mastery_tier, core_score,
   clinical_score, reach_status, hint_level_final, turn_count, locked_*,
   status). Note: row already exists from session-start insert per L21
   (Codex fix #2) — this is an UPDATE, not INSERT.
3. Persist full message log (now including the closing message from step 1)
   to data/student_state/sessions/{thread_id}.json
4. Upsert subsection_mastery row (EWMA blend per L3)
5. Skip steps 6-7 if pre-lock terminated (no transcript to extract from)
6. Run ONE Haiku extraction call:
   - Input: full transcript (with closing message included) + locked_question
     + locked_answer + reach_status + image_session flag (true if image-initiated)
   - Output: { misconception_observations: [str, ...],
               learning_style_notes: [str, ...],
               key_takeaways: { what_demonstrated: str, what_needs_work: str } }
7. For each extracted misconception/learning_style item: safe_mem0_write with
   required metadata (subsection_path, section_path, session_at, thread_id,
   category) per L4. Note: topic_path REMOVED per Codex fix #4.
8. Store key_takeaways in sessions.key_takeaways JSON column
9. END
```

Cost per session-end: 1 Teacher closing draft (~$0.005) + 1 Haiku extraction (~$0.001) + 2-5 mem0 writes (~$0.0001 each) = ~$0.007.

### Terminal close ownership rule (Codex round-2 fix)

Multiple terminal nodes can compose closing messages (L22 cap-7 farewell, L56 off-domain end, L58 deflection confirm, L65 clinical opt-out, L73 confirm-and-end, L26 turn-25 graceful close). To avoid double-rendering or ambiguity:

**Rule: each terminal NODE composes its own closing message via Teacher with the appropriate mode/tone combination.** Examples:

| Terminal path | mode | tone |
|---|---|---|
| Turn-25 graceful close (L26) | `honest_close` | `honest` |
| Off-domain strike-4 end (L56) | `confirm_end` | `firm` |
| Deflection confirm-and-end (L58) | `confirm_end` | `neutral` |
| Cap-7 give-up farewell (L22) | `honest_close` | `neutral` |
| Clinical opt-out close (L65) | `honest_close` | `encouraging` |
| Natural reach + clinical-skip (L65) | `honest_close` | `encouraging` |

The composed message is appended to `state.messages` BEFORE control passes to `memory_update_node`.

**`memory_update_node` (L63) does NOT compose a redundant closing message.** Its step 1 is conditional:
- If terminal node already appended a closing message → skip step 1 (close already in state.messages)
- If terminal node didn't (rare — direct memory_update entry on natural reach without explicit close) → compose the standard `mode="honest_close", tone="honest"` summary

**Closing-message marker shape (Codex round-3 fix #3 — backend-state only, NOT a frontend chat API field):**

```python
state.messages.append({
    "role": "tutor",
    "content": "...the actual closing prose...",
    "phase": "memory_update",
    "metadata": {"is_closing": True}
})

# memory_update detection logic:
last_msg = state.messages[-1] if state.messages else {}
already_closed = (
    last_msg.get("role") == "tutor"
    and last_msg.get("metadata", {}).get("is_closing") is True
)
if not already_closed:
    # compose standard honest_close (mode="honest_close", tone="honest")
    ...
```

Frontend renders `content` like any other tutor message — the `metadata.is_closing` flag is only consumed by backend orchestration. No new chat API surface needed.

Result: exactly one closing message per session, owned by whichever node terminated. memory_update is responsible for persistence + extraction only, not composition (unless no one composed yet).

# 📚 INGESTION + RAG — Locked Decisions (L76)

Added 2026-05-02 after auditing missing-metadata + orphan-chunk issues in the corpus.

## L76. Ingestion Metadata Cleanup — LLM-Based, No Orphans

**Hard invariant:** every chunk in `data/processed/chunks_openstax_anatomy.jsonl` MUST have all three hierarchy levels populated — `chapter_title`, `section_title`, `subsection_title`. Validation gate at end of ingestion fails the build if any chunk lacks any field.

No orphaned chunks. No heuristic backfills. Only LLM-based recovery.

### Algorithm

Bundled into `scripts/build_topic_index.py` (alongside L19 title rewrites, L38 summary coverage, L40 trigger):

```
For each chunk in raw chunks pool:

1. VALIDATE pass — classify completeness:
   - "complete" = all 3 levels present
   - "incomplete_section" = chapter present, section missing
   - "incomplete_all" = no section/chapter at all
   - "section_level" = chapter + section present, subsection missing

2. If "complete" → keep as-is, tag source="parser"

3. If incomplete (any kind):
   a. Haiku attempt:
      - Inputs: chunk text + topic_index TOC + raptor summaries + neighbors (preceding + following chunk text, 200 chars each)
      - Output: { action: "use_existing" | "create_new", chapter, section, subsection, confidence, rationale }
      - Confidence ≥ 0.8 (use_existing) OR ≥ 0.85 (create_new) → accept

   b. If Haiku confidence below threshold → escalate to Sonnet:
      - Inputs: above + entire section text + all sibling subsections (with summaries) + 500 chars preceding/following
      - Confidence ≥ 0.8 → accept (lower threshold OK — Sonnet has fuller context)

   c. If Sonnet still uncertain → FORCE create_new at Sonnet's best guess:
      - No drops, no orphans (per user invariant)
      - Tag with source="llm_synthesized" + low_confidence flag for audit

4. For "create_new" decisions (Haiku-detected or Sonnet-forced):
   - Add new entry to textbook_structure.json with source="llm_synthesized"
   - Generate raptor summary (Sonnet)
   - Generate display_label (Sonnet, per L19)

5. After all chunks processed, VALIDATION GATE:
     for chunk in all_chunks:
         assert chunk.chapter_title, f"Missing chapter_title: {chunk.chunk_id}"
         assert chunk.section_title, f"Missing section_title: {chunk.chunk_id}"
         assert chunk.subsection_title, f"Missing subsection_title: {chunk.chunk_id}"
   Build fails if any chunk lacks a field.
```

# 🖼️ VLM (IMAGE INPUT) — Locked Decisions (L77)

Added 2026-05-02 — image-driven topic entry for OT (anatomy) sessions.

## L77. VLM Flow — Image-Driven Topic Entry + Image-Focused Tutoring

**Scope:** student can upload an anatomical image (X-ray, diagram, histology slide, dissection photo, anatomical model photo) at the rapport / pre-lock phase. The session adapts to focus on identifying structures within the image, not just generic tutoring on the mapped topic.

### When image upload is available

- **Pre-lock only.** Image upload appears as a button in the rapport UI alongside the input box and topic cards.
- **Not mid-tutoring** (deferred — adds state complexity, ship simple version first).
- **Not in clinical phase** (clinical scenario is text-driven via Dean per L74).

### Upload UX

- Upload button → file picker → image stored at `data/uploads/{thread_id}/{image_id}.{ext}`
- Single image per upload. **One retry allowed** — if VLM produces low-confidence JSON OR topic_mapper fails to map, UI shows "Retry with different image" + "Type topic instead" buttons.
- File size cap: 5 MB. Format: PNG / JPG / WebP.
- Loading state during VLM call (Sonnet vision ~3-5s).

### VLM call (one-shot, Sonnet vision)

Single call at upload time. Subsequent Teacher turns use the extracted text — no per-turn vision calls (cost discipline + simpler architecture).

**VLM output JSON schema:**
```json
{
  "identified_structures": [
    {"name": "humerus", "location": "central, vertical", "confidence": 0.95},
    {"name": "deltoid muscle", "location": "lateral, top", "confidence": 0.88},
    {"name": "glenohumeral joint", "location": "center top", "confidence": 0.91}
  ],
  "image_type": "x-ray" | "diagram" | "histology" | "gross_anatomy" | "model" | "other",
  "description": "X-ray of right shoulder in AP view, showing humerus, glenoid fossa, deltoid muscle, and acromioclavicular joint",
  "best_topic_guess": "shoulder anatomy",
  "confidence": 0.9
}
```

Cost: ~$0.003 per upload. Trivial.

### Topic-mapping path

VLM output's `best_topic_guess` + `description` feed into `topic_mapper_llm` (L9) as if student typed that text:
- topic_mapper sees the description as student input
- emits standard `{verdict, confidence, top_matches}` output
- Routes per L9 thresholds:
  - `verdict=strong` → lock to that subsection
  - `verdict=borderline` → cards (with image still attached for context)
  - `verdict=none` → retry button OR fallback cards

If JSON parse fails OR `confidence < 0.7` OR `identified_structures` is empty → trigger retry UI.

### Image-focused tutoring (the differentiator)

Once locked, Dean's `lock_anchors_call` receives the VLM JSON as additional input. Dean generates an IMAGE-SPECIFIC `locked_question`:

| Generic tutoring | Image-driven tutoring |
|---|---|
| "What are the three joints in the shoulder?" | "Looking at your X-ray, what bone forms the central vertical structure?" |
| "What's the function of the deltoid?" | "In the image you uploaded, identify the muscle visible on the lateral side near the top." |
| "How does the glenohumeral joint work?" | "Trace the structures forming the joint visible at the center top of your image." |

Hint plan adapts to identification scaffolding pattern: *"what shape is it?"* → *"what color/density is it?"* → *"where does it attach to the surrounding structures you can see?"* → identification.

### TurnPlan extension

VLM JSON persists in TurnPlan throughout the session as `image_context` field:

```typescript
TurnPlan {
  ...all existing fields per L46...
  image_context: dict | null   // VLM JSON output, present when session was image-initiated
}
```

Teacher uses `image_context.description` + `identified_structures` to ground every turn's response. No vision capability needed at Teacher level — text description is sufficient.

### Updates to related decisions

- **L9** (topic mapper) — accepts VLM JSON description as alternative input; same output schema
- **L46** (TurnPlan schema) — extends with `image_context: dict | null`
- **L23** (anchor question repetition) — same student × subsection × image yields different angles; lock prompt also receives image_context to differentiate "what is structure A" from "what attaches to structure A"

### Lock prompt instruction (gap-close 1)

When `image_context` is present in TurnPlan input to `lock_anchors_call`, the lock prompt receives an explicit instruction:

> *"If `image_context` is present, generate a `locked_question` that REQUIRES reading the image — identification, structural relationships, or what-do-you-see questions. Do NOT generate generic textbook-recall questions when an image is available."*

Without this explicit instruction the LLM may default to generic recall.

### Forbidden_terms for image sessions (gap-close 2)

`forbidden_terms` for image sessions includes:
- `locked_answer` (existing)
- aliases (existing)
- Other VLM-identified structures the student is meant to discover later (NEW)

Rationale: if locked_answer is "humerus" and the same image shows the deltoid, Teacher mentioning "the deltoid attaches above this bone" gives away contextual hints. Forbid mentioning sibling identified structures unless they're the current focus.

The forbidden_terms list is generated by Dean at lock time using `image_context.identified_structures` minus the structure currently being identified.

### Fallback cards leverage VLM-identified structures (gap-close 3)

When topic_mapper returns `verdict=none` for VLM-derived input (rare — happens when image content is real anatomy but doesn't map cleanly to any TOC node), the fallback cards are smart, not random:

- `sample_diverse` (per L11/L22 pattern) gets a similarity hint: VLM's `identified_structures` list
- Cards surfaced are those whose subsections relate to the identified structures
- Falls back to fully random `sample_diverse` only if no related TOC nodes found

UX: student sees *"Topics related to your image: Shoulder Joint, Glenohumeral Anatomy, Rotator Cuff"* — relevant to what they uploaded.

### Eval scorer extension (gap-close 4)

L39 (eval scorer updates) extends with: scorer reads `state.image_context` alongside `state.anchor_chunks` + `state.tangent_chunks` for RAGAS context_precision/recall computation. Image description counts as a grounding source for Teacher's responses on image-driven sessions.

Without this, scorer marks image-grounded responses as "ungrounded" because chunks don't contain the image content.

### Domain gating

Entire VLM flow is gated by `cfg.domain.vlm.enabled`. Set per domain:
- **OT**: `vlm.enabled: true` — full image upload + VLM + image-focused tutoring per this section
- **Physics**: `vlm.enabled: false` — no image upload at all. Frontend hides upload button. Backend rejects upload requests with HTTP 403. VLM never called.

Single flag controls the entire surface area (UI + backend + flow). No partial enablement.

### Flow integration — no new graph node

Image upload is handled at the **backend / WebSocket layer** as a pre-graph operation. The graph topology stays unchanged.

```
On image upload (WS message type=image):
  1. Backend stores image at data/uploads/{thread_id}/{image_id}.{ext}
  2. Sonnet vision call → VLM JSON
  3. Validate (parse OK + identified_structures non-empty + confidence ≥ 0.7)
     - Fail → return retry UI to client, graph NOT invoked
     - Pass → continue
  4. Inject into state:
       state.image_context = vlm_json
       state.messages.append({role: "student", content: vlm_json.description, image_id: ...})
  5. Invoke graph normally (rapport_node skips itself,
     dean_node sees image_context, topic_mapper processes the description)
```

VLM call logged to `state.debug.turn_trace` by the backend wrapper before invoking graph, so it appears in session export alongside other LLM calls.

No new graph node, no conditional edges. Same `dean_node` handles image-derived input identically to text-derived input — the only difference is `state.image_context` being populated.

If image upload ever extends mid-session (deferred), the same backend pattern applies — just inject `image_context` mid-conversation.

### Image retention + cleanup

Original image is **kept** at `data/uploads/{thread_id}/{image_id}.{ext}` after the session.

Used by:
- **My Mastery revisit chat** (per L31) — when student revisits an image-initiated session, the original image is re-displayed in the analysis view alongside the transcript. Path stored in `sessions.image_path` (new column on the SQL `sessions` table per L2).
- **Future re-analysis** — if the student returns and wants to discuss the same image in a fresh tutoring session, image is available.

Cleanup policy: purge after 30 days OR when student deletes session (separate ops policy, not architectural).

### Cost cap

Hard cap: 5 image uploads per session. Above the 1-retry allowance for misclassification edge cases, but bounds runaway cost (5 × ~$0.003 = $0.015 max VLM cost per session).

If student tries to upload a 6th, UI rejects: *"Upload limit reached for this session. Continue with current topic or start a new session."*

### Safe VLM wrapper (gap-close A)

Mirror the mem0 safety pattern (L5):

```python
def safe_vlm_call(image_path: str, state: dict) -> dict | None:
    """Always returns dict OR None. Never raises. Logs to state.debug.turn_trace."""
    started = time.time()
    err = None
    result = None
    try:
        result = sonnet_vision_call(image_path)  # the actual VLM call
    except Exception as e:
        err = str(e)
    elapsed_ms = int((time.time() - started) * 1000)
    _trace_vlm(state, op="vlm_call", image_path=image_path,
               result_summary=summarize(result), elapsed_ms=elapsed_ms, error=err)
    return result
```

Failure handling at backend:
- Returns None → backend shows generic *"Couldn't process the image — try again or type your topic"* + retry button
- Backend treats this same as low-confidence retry (Q-VLM-2 pattern)
- Trace entry recorded for session export

Three guarantees: never crashes graph, distinguishes parse-fail vs infra-fail in trace, every VLM op visible in session export.

### Confidence threshold semantics (gap-close B)

Two confidence values in the VLM JSON:
- **Top-level `confidence`** = overall image identifiability ("how confidently can I describe this image?")
- **Per-structure `confidence`** = how sure VLM is about each individual structure

**Threshold rule:**
- Top-level `confidence ≥ 0.7` is the gate for "accept the JSON, proceed to topic_mapper". Below → retry UI.
- Per-structure confidences are informational. At Dean's lock-anchor step, only structures with per-structure `confidence ≥ 0.7` are eligible for `forbidden_terms` enumeration and identification-question targets. Lower-confidence structures are listed in `image_context.identified_structures` but not used as gating signals.

This lets a generally-clear image with one ambiguous structure still proceed (top-level confident), while keeping uncertain structures out of leak-prevention forbidden lists (where false-positives matter more).

### Text + image hybrid input (gap-close C)

If student uploads image AND types text in the same submission (*"I uploaded this X-ray, what's the bone in the middle?"*):

**Topic_mapper input strategy:**
- If student typed substantive text (≥ 3 words): use student text as primary topic_mapper input. `image_context` still populates from VLM for downstream image-focused tutoring.
- If student typed nothing OR < 3 words: use VLM `description` as topic_mapper input.

Student's typed intent is more authoritative than VLM's auto-generated description when both are present. Image still informs `image_context` for the locked_question style.

### Minor production-readiness notes

- **Image security:** out of scope for thesis (single-user). Production hardening TODO: upload endpoint requires auth, image file paths scoped to student_id, served via authenticated endpoint not direct file access. Add TODO comment in upload handler.
- **MIME validation:** ship it. Backend validates uploaded file via magic-byte check (not just extension). Reject anything that doesn't sniff as image/png, image/jpeg, image/webp. Standard library — `python-magic` or stdlib `mimetypes` + signature check.
- **Memory writes for image sessions:** L63's session-end extraction Haiku call receives `image_session: True` flag in input when applicable. Allows extracted misconception/learning_style entries to capture visual-identification patterns (*"struggled to identify deltoid in X-ray view"*, *"responded well to identification scaffolding"*). Same extraction pipeline, just enriched context.

### What's deferred

- Mid-session image upload (post-lock tangent retrieval with image) — defer until eval shows demand
- Multiple-image stacked sessions (3 views of same structure in one session) — defer
- Per-turn vision capability — defer; one-shot VLM + text description is sufficient for v1

# 🔊 ACCESSIBILITY — Locked Decisions (L79)

Added 2026-05-02 — TTS + STT for accessibility bonus per project spec.

## L79. Accessibility — Browser-Native TTS + STT (Bonus)

**Pure frontend feature. Zero backend changes.**

### TTS (Text-to-Speech) — tutor messages read aloud

- Frontend toggle in user settings: `tts_enabled` (default: false)
- When toggle is on, every tutor message that arrives via WS is also passed to browser's `SpeechSynthesis` API (Web Speech API native)
- Voice selection from browser's available voices (default: en-US neutral)
- User can pause / resume / skip via small floating playback controls
- Streaming text → speech sync: speak chunks as they arrive (don't wait for full message), pause if streaming pauses

Implementation: ~50 lines of React in `frontend/src/components/chat/MessageBubble.tsx` + a settings toggle. No new dependencies (Web Speech API is browser-native).

### STT (Speech-to-Text) — student speaks responses

- Microphone button next to chat input
- Click → browser activates `SpeechRecognition` API (Web Speech API native)
- Recording indicator while active
- Transcribes speech → fills chat input field (student can edit before sending)
- Click again or press send → submits as standard WS text message
- No auto-send on transcription end (student controls submission)

Implementation: ~80 lines of React in `frontend/src/components/chat/InputBox.tsx`. No new dependencies.

### Browser support

Web Speech API is supported in Chrome, Edge, Safari. Firefox has partial support. UI gracefully degrades — if API unavailable, hide TTS toggle + STT mic button.

### What's NOT in this decision

- Server-side TTS (ElevenLabs, OpenAI Whisper, etc.) — defer; browser-native is free + simple + good-enough for demo
- Multilingual support — defer; English only for thesis
- Voice cloning / personality — defer
- Real-time streaming STT during long monologues — defer; click-to-record-then-stop pattern is sufficient

### Cost

Zero. Browser-native APIs are free, run client-side, no server load.
